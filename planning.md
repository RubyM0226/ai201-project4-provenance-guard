# Provenance Guard — Planning

## 1. Detection Signals

**Signal 1 — LLM-based classification (Groq, llama-3.3-70b-versatile).**
Sends the submitted text to the model with a prompt asking it to judge, holistically,
whether the text reads as human-written or AI-generated. Output: a JSON object with
`ai_likelihood` (float 0–1) and a one-sentence `reasoning` string. This captures semantic
and stylistic coherence as a whole — the kind of judgment a careful human reader would
make, including things like generic reasoning patterns or overly tidy argument structure
that no word-count-based metric can see.

**Signal 2 — Stylometric heuristics (pure Python).**
Computes three measurable properties: sentence-length variance, type-token ratio
(vocabulary diversity), and punctuation density. Combines them into a single 0–1 score
where low variance + low vocabulary diversity + moderate-to-high transitional
punctuation pushes the score toward "AI-like." Output: a float 0–1 plus the raw metrics
for transparency.

**Combination:** weighted average, `confidence = 0.65 * llm_score + 0.35 * stylometric_score`.
The LLM signal gets more weight because it reasons about meaning; stylometrics is treated
as a corroborating structural check, not the primary signal, since it's the noisier of
the two on short text (see Anticipated Edge Cases below).

## 2. Uncertainty Representation

A confidence score is the weighted combination above, always a float in [0, 1]. It is
**not** a probability in the statistical sense — it's a calibrated blend of two heuristic
signals, and it's documented as such rather than oversold as ground truth.

Thresholds (intentionally asymmetric — see rationale):
- `confidence >= 0.70` → **likely_ai**
- `confidence <= 0.35` → **likely_human**
- otherwise → **uncertain**

**Why asymmetric:** a false positive (telling a real human creator their work was flagged
as AI) is worse than a false negative on a writing platform — it's the failure mode that
damages trust and drives creators away. So the system requires a much higher bar of
combined evidence (0.70+) before committing to the "likely_ai" label, while a comparatively
lower bar (0.35-) is enough to commit to "likely_human." The gap between 0.35 and 0.70 is
the "uncertain" band — deliberately wide, because an honest "we don't know" is safer than a
confident wrong answer in either direction.

A 0.51 score and a 0.95 score both land in different buckets by design: 0.51 is deep in
"uncertain" (hedged, appeal-forward language); 0.95 is deep in "likely_ai" (direct language,
still appealable).

## 3. Transparency Label Design

Exact text of the three variants (also in README.md, as required):

| Variant | Text |
|---|---|
| High-confidence AI | "Likely AI-Generated — Our system detected strong signals that this content was generated by an AI model. If you believe this is incorrect, you can appeal this classification." |
| High-confidence human | "Likely Human-Written — Our system found this content consistent with human authorship." |
| Uncertain | "Uncertain — Our system could not confidently determine whether this content is AI-generated or human-written. Treat this result as inconclusive. You can appeal this classification for human review." |

Each is appended with `(confidence score: X.XX)` at render time so the number is always
visible alongside the plain-language verdict.

## 4. Appeals Workflow

- Any creator can appeal any classification of their own content by POSTing `content_id`
  and `creator_reasoning` to `/appeal`.
- On receipt: the system looks up the content_id in the audit log, sets `status` to
  `"under_review"`, records the `creator_reasoning` and an `appeal_timestamp` on that same
  row (not a separate table — keeps the full history of one piece of content in one place).
- A human reviewer opening the appeal queue (in a real deployment, a filtered view of
  `GET /log?status=under_review`) would see: the original text preview, both signal scores,
  the combined confidence, the label that was shown to the reader, and the creator's
  reasoning — everything needed to make a call without re-running detection.
- Automated re-classification is explicitly out of scope for this project.

## 5. Anticipated Edge Cases

1. **Short or thin text (under ~10 words / 2 sentences).** The stylometric signal returns
   a neutral 0.5 with a `confidence_in_signal: "low"` flag rather than a real score, because
   variance and type-token ratio are statistically meaningless on a handful of words. A
   two-line poem or a one-sentence caption will lean entirely on the LLM signal.

2. **Formal, jargon-heavy human writing (e.g. academic or technical prose).** A human
   economist writing about monetary policy uses long, structurally uniform sentences and
   domain vocabulary that repeats — the same surface features the stylometric heuristic
   associates with AI generation. This signal cannot distinguish "polished/formal" from
   "generated"; it only measures uniformity. This is a real, tested case (see README) and
   is the main reason stylometrics is weighted below the LLM signal rather than equally.

3. **Text with heavy repetition or simple vocabulary used deliberately for style** (e.g. a
   poem built on repetition, a children's-book-style piece). Low type-token ratio here
   reflects an artistic choice, not AI generation, and will pull the stylometric score up
   even though the fit is not causal — vocabulary repetition means "AI-like" only as a
   correlation, not a rule.

## Architecture

```
                         SUBMISSION FLOW
                         ----------------
   client
     |
     |  POST /submit  { text, creator_id }
     v
+-------------+      +------------------+      +------------------------+
|  Flask app  |----->|  signals.py      |      |                        |
|  /submit    |      |  llm_signal()    |----->|  scoring.py            |
|  route      |      |  stylometric_    |----->|  combine_confidence()  |
+-------------+      |  signal()        |      |  -> {confidence,      |
      |               +------------------+      |     attribution}      |
      |                                          +------------------------+
      |                                                     |
      |                                                     v
      |                                          +------------------------+
      |                                          |  labels.py             |
      |                                          |  generate_label()      |
      |                                          +------------------------+
      |                                                     |
      v                                                     v
+---------------------------+                    response: {content_id,
|  audit_log.py              |<-------------------  attribution, confidence,
|  log_submission()          |                       label, signals}
|  (SQLite: audit_log table) |
+---------------------------+


                         APPEAL FLOW
                         -----------
   client
     |
     |  POST /appeal  { content_id, creator_reasoning }
     v
+-------------+      +---------------------------+
|  Flask app  |----->|  audit_log.py              |
|  /appeal    |      |  log_appeal()              |
|  route      |      |  - find row by content_id  |
+-------------+      |  - status -> under_review  |
      |               |  - store reasoning +       |
      |               |    timestamp               |
      v               +---------------------------+
  response: {content_id, status: "under_review", message}
```

**Submission flow narrative:** a client posts raw text; the Flask route calls both
signal functions independently, passes their outputs into the scoring function to get a
single confidence + attribution bucket, generates the label text for that bucket, writes
one structured row to the SQLite audit log, and returns everything (content_id, verdict,
confidence, label, and both raw signal outputs) to the client in one response.

**Appeal flow narrative:** a client posts a content_id plus their reasoning; the route
looks up the existing audit log row for that content_id, updates it in place (status,
reasoning, appeal timestamp) rather than creating a new record, and confirms receipt.
`GET /log` surfaces the full history either way.
